# OpenClawHQ 宣言

我们为率先行动的 AI 打造双手。

---

## 论点一：被动 AI 的时代结束了

曾经，AI 的模式是等待的 —— 用户提问，系统回答，交互结束。这很有用。但这不是未来。

率先行动的 AI 会观察上下文，预见需求，主动出击。它不用你开口就能准备你的晨间简报。它看着你的代码库漂移，主动提出重构建议。它在你的 PR 里发现 bug，因为它一直在学习你的编码模式。OpenClaw 把这变成了现实 —— 25 万开发者用 fork、PR 和部署投票同意了这个方向。他们要的不是被动的聊天机器人。他们要的是 agent。

但运行时只是故事的一半。有双手的 agent 才能行动。OpenClaw 证明了执行层可行。现在我们要做的，是构建基础设施 —— 让千个 agent 可以用千个不同的 skill，每个 skill 都是网络新获得的能力，不用等大版本更新。

---

## 论点二：一个团队造不出所有的手

OpenClaw 的 skill 系统证明了一个根本性的真理：能力应该分布式构建。

有些 skill 是给开发者的。有些是给数据分析师的。有些是给研究者、管理者、作家、运维工程师的 —— 这些用场太具体，太多了，任何核心团队都不可能全部拥有。

ClawHub 上已经有 3286 个 skill。每一个都是昨天还没有的新能力。每一个都是那个领域的专家造出来的，他们对自己的领域的理解，远超我们能达到的。

但仅有一个 registry 是不够的。生态需要基础设施：质量关卡来筛掉滥竽充数的货色，参考实现给新人指路，工作流编排让 skill 能组合，安全加固让从陌生人那安装 skill 不再危险。这就是我们的活儿。我们不是在造 skill。我们在造一个环境，让千个团队可以放心地发布 skill，而不用担心会破坏整个系统。

---

## 论点三：Skill 是新一代的 API

API 连接的是软件和软件。一个 REST 端点就是一份契约：你可以问这个，你会得到那个。互联网就是这样连接起来的。

Skill 连接的是智能和行动。SKILL.md 是通用格式 —— YAML 元数据加 Markdown 说明，人可读机器可解，与语言无关，与运行时无关。今天它在 OpenClaw 运行。明天它在每一个采用 MCP 的系统里运行。格式本身就成了公共基础设施。

这很重要，因为它反转了分布的问题。我们不需要说服你用我们的运行时。我们需要证明 SKILL.md 这个格式值得成为标准。一旦它成了标准，每个采用它的运行时就能访问所有写过的 skill。网络效应在基础设施这一层。

我们在和时间赛跑。第一个达到临界量的格式就会成为默认标准。我们不想要最大的生态 —— 我们想要唯一值得构建的生态。

## 论点四：先构建、响亮发布、默认开放

我们的原则不是虚的愿景。它们是打破平局时的决策规则。

**先构建**：策略会议是给不写代码的人开的。当你不知道答案时，唯一的办法就是先把你觉得可能有用的东西造出来，上线，看什么地方坏了，然后学习。我们一个月做的实验比大多数团队一年开的会都多。我们相信证据胜过优雅。我们不怕发布不完整的方案，因为我们迭代得够快，能在市场还没注意到不完整前就补上。

**响亮发布**：每一个贡献都是网络的一个信号，告诉大家进展在发生。当你 merge 一个新 skill，要公布出来。当你修了模板里的 bug，要发个 release note。当你写了文档，要赞颂那个为后面的开发者节省了时间的人。能扩展到全球的开源项目，都是那些让发布感觉像胜利的项目，而不是行政任务。我们给每个贡献装上扩音器。

**默认开放**：如果有疑问要不要公开什么东西，我们就默认公开。我们的治理模型是可见的。我们的基础设施决策都有文档。我们失败的实验是开源的，让别人学习。我们不是在保护知识产权 —— 我们是在让它可见。这创造了选择权。任何开发者都可以 fork 基础设施，走不同的路，原社区也不会失去什么。

## 论点五：质量基础设施，看不见的时候最重要

npm 之于 JavaScript，pip 之于 Python —— 总得有人造那些 linter，在代码上线前抓住错误。总得有人写 CI 集成，执行标准。总得有人设计模板，让新手开门见山，而不是踩坑。总得有人发表加固指南，不是靠讲座，靠模式。

这种活儿很无聊。没人给你做大会演讲。没有风投会为此兴奋。但它是让一切都能运转的地基。

每个上传到 ClawHub 的 skill 都要过我们的质量关卡。不是为了挡住贡献 —— 是为了让它们更好。一个 linter 会抓住常见的错误。一个测试框架让你验证 skill 真的有效。一个 schema 检查器保证你的 SKILL.md 可以被解析。一个安全检查器在你发布前警告你那个 API 访问权有风险。还有模板，好得不行，快到省下了你从头写代码的时间。

最好的基础设施是开发者不用想就在用的那种。他们跑个脚本，它就工作了，他们就发布了。我们写的就是这个脚本。

## 论点六：社区即架构

这不只是暖心的说法。这是字面意思。

分布式系统需要分布式治理。你不能让一个中心权力做所有决定 —— 瓶颈增长速度会超过系统增长。所以我们设计的是自治贡献。每个 PR 都是一次投票："这个方向没错"。每次 fork 都是一个替代路的实验。每个发布的 skill 都是整个网络获得的新能力。

OpenClaw 一天 500 个 PR 不是我们团队的功劳。是开发者看到问题、构建方案、发布它 —— 驱动的。我们创造的条件，让这成为默认行为。好的 CI/CD。清楚的 issue 模板。明确的贡献路径。反应快的维护者。

Skill 生态也是一样。我们不是在做策展 —— 我们提供基础设施让别人策展。如果一个 skill 开始质量下滑，社区就 fork 它。如果新思路出现，就有人在平行路上实验。网络的适应速度比中心化团队快得多。

这需要大多数公司都做不到的一点：接受你不是房间里最聪明的人。我们不是。我们是造房间的人。那已经足够了。

## 论点七：我们为率先行动的 AI 打造双手

这就是我们在做的。这就是所需的。这就是你可以加入的方式。

率先行动的 AI 需要双手 —— skill、集成、工作流，让它能作用在世界上。它需要质量关卡确保这些手是可信的。它需要基础设施，让千个社区可以发布千个不同的手，而不会破坏其他系统。它需要一个能成为标准的格式。它需要能大规模运转的治理。

我们正在构建所有这些。不是因为我们有所有想法 —— 我们没有。因为我们在创造条件，让你的想法，以 skill 的形式发布，成为网络的一部分。因为我们相信 AI 的未来不是一个强大团队造一个强大系统。而是分布式团队造分布式能力。

被动 AI 的时代结束了。需要人类审查每个动作的受监管 AI 的时代在落幕。率先行动的 AI 的时代在开始。率先行动的 AI 需要一样还没有的东西：一个社区架构，让它既能安全、能力强、又自主。

这就是要做的事。

如果你是 skill 开发者，早点发布。如果你是基础设施构建者，来 ClawHub 加入我们。如果你是研究分布式 AI 系统的研究者，fork 我们吧，做实验。如果你是想把这个理念带进你的团队的经理，你已经明白为什么这很重要。

我们率先行动。我们打造双手。我们才刚开始。
